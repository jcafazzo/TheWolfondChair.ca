  <name>Task 1: Replace viewport units and add touch/overscroll CSS properties</name>
  <files>wolfond-report-2024-2026.html</files>
  <action>
Make these exact CSS edits in the `<style>` block:

**MCSS-01 — Viewport height fallback (3 locations):**

1. **`body` rule (around line 34):** Replace `height: 100dvh;` with two declarations:
   ```css
   height: 100vh;
   height: 100svh;
   ```
   The first line is the fallback for browsers without `svh` support. The second overrides it in modern browsers. Both lines must be present — cascade override pattern.

2. **`.deck` rule (around line 40):** Replace `height: 100dvh;` with:
   ```css
   height: 100vh;
   height: 100svh;
   ```

3. **`.slide-inner` rule (around line 65):** Replace `min-height: 100dvh;` with:
   ```css
   min-height: 100vh;
   min-height: 100svh;
   ```
   Note: this is `min-height`, not `height`. The property name stays the same — only the unit value changes.

**CRITICAL — DO NOT CHANGE these other `dvh` occurrences:**
- `.scroll-content` `max-height: calc(100dvh - 10rem)` — intentionally uses dvh (responds to current viewport state)
- Inline `style` attributes on `.scroll-content` divs (lines ~666, ~715) — out of scope per user decision
- `.split-image` `min-height: 100dvh` (line ~164) — overridden by media query on mobile, out of scope

**MCSS-02 — Add `touch-action: pan-y` to `.slide` rule (around line 42):**
Add this property to the existing `.slide` rule:
```css
touch-action: pan-y;
```
This tells the browser to handle vertical panning natively while passing horizontal touch events to JS. This ONLY works because the existing touchstart listener uses `{ passive: true }` — do NOT add `touch-action` to any child element (no `.slide-inner`, no `.scroll-content`). Apply ONLY to `.slide`.

**MCSS-04 — Add `overscroll-behavior-x: contain` to `.slide` rule (same location):**
Add this property to the same `.slide` rule alongside `touch-action`:
```css
overscroll-behavior-x: contain;
```
This prevents horizontal swipe overscroll from triggering browser back/forward navigation. Known limitation: iOS Safari < 16 ignores this property — accepted for Phase 1 per STATE.md blocker note.

Also add `-webkit-tap-highlight-color: transparent;` to the `.slide` rule to suppress the default iOS tap highlight flash.

**MCSS-03 — Nav dot touch target expansion:**

1. **Add `position: relative` to `.slide-dot` rule (around line 93):**
   The existing `.slide-dot` rule sets `width: 8px; height: 8px;` and various other properties. Add `position: relative;` to it — this is required for the absolutely-positioned `::before` pseudo-element.

2. **Add new `.slide-dot::before` rule immediately after the `.slide-dot` rule:**
   ```css
   .slide-dot::before {
     content: '';
     position: absolute;
     top: 50%;
     left: 50%;
     transform: translate(-50%, -50%);
     width: 44px;
     height: 44px;
   }
   ```
   This creates an invisible 44px x 44px tap zone centered on each 8px dot. No background color — purely a hit area expansion. The `::before` pseudo-element approach preserves the exact visual spacing between dots (unlike padding, which would change the gap in the flex column layout).

3. **Add `-webkit-tap-highlight-color: transparent;` to `.slide-dot`** to suppress the default iOS tap highlight that would flash over the expanded invisible area.

4. **Verify arrows are already sufficient:** The `.slide-arrow` elements are already 48px x 48px (exceeds 44px minimum). No changes needed for arrows. But DO add `-webkit-tap-highlight-color: transparent;` to `.slide-arrow` as well for consistent tap feedback.

**IMPORTANT:** The `.slide-nav` parent has `mix-blend-mode: difference` (around line 90). The `::before` pseudo-element has no background, so the blend mode does not affect it visually. If testing reveals any ghost artifact, add `mix-blend-mode: normal;` to `.slide-dot::before`.
