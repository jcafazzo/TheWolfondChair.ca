**The bug:** At touchend (around line 1247), the code does `const scrollEl = slides[current].querySelector('.scroll-content')` to check if the current slide has scrollable content. This queries the DOM at touchend time using the *current slide index*, but by touchend the finger may have moved — the scroll element should be captured where the finger *landed* at touchstart, not where the DOM says .scroll-content is at touchend.

**The fix — 3 changes in the `<script>` block:**

1. **Add `touchScrollEl` variable declaration (around line 1233):**
   Find the existing `let touchStartY = 0, touchStartX = 0;` declaration. Change it to:
   ```javascript
   let touchStartY = 0, touchStartX = 0, touchScrollEl = null;
   ```

2. **Capture scroll element at touchstart (around lines 1234-1236):**
   Find the existing touchstart listener:
   ```javascript
   document.addEventListener('touchstart', (e) => {
     touchStartY = e.touches[0].clientY;
     touchStartX = e.touches[0].clientX;
   }, { passive: true });
   ```
   Add one line inside the handler to capture the scroll element:
   ```javascript
   document.addEventListener('touchstart', (e) => {
     touchStartY = e.touches[0].clientY;
     touchStartX = e.touches[0].clientX;
     touchScrollEl = e.target.closest('.scroll-content');
   }, { passive: true });
   ```
   **CRITICAL:** The `{ passive: true }` option MUST remain unchanged. This is required for `touch-action: pan-y` (from Plan 01) to work. Do NOT change to `passive: false`.

3. **Use captured `touchScrollEl` in touchend handler (around lines 1245-1249):**
   Find the section in the touchend handler that checks for scrollable content. The existing buggy code looks like:
   ```javascript
   const scrollEl = slides[current].querySelector('.scroll-content');
   if (scrollEl && scrollEl.scrollHeight > scrollEl.clientHeight + 4) return;
   ```
   Replace with:
   ```javascript
   if (touchScrollEl && touchScrollEl.scrollHeight > touchScrollEl.clientHeight + 4) return;
   ```
   Delete the `const scrollEl = slides[current].querySelector('.scroll-content');` line entirely — it's replaced by the `touchScrollEl` captured at touchstart.

**Why MJS-01 is covered:** The existing 60px threshold and `Math.abs(dx) > Math.abs(dy)` horizontal-vs-vertical discrimination (around lines 1240-1244) is already correct. MJS-01 requires "horizontal delta exceeds vertical delta and minimum 40px threshold" — the existing code uses 60px which exceeds the 40px minimum. The swipe logic works correctly once MJS-02's bug is fixed, because the scroll detection no longer incorrectly suppresses vertical swipes.

**Preserved behavior:**
- Horizontal swipe (dx > 60, dx > dy): advance/retreat slide — unchanged
- Vertical swipe on non-scrollable slide (dy > 60, dy > dx, no scroll content): advance/retreat slide — unchanged (user decision: keep this intuitive behavior)
- Vertical swipe on scrollable slide: scroll content, don't advance — NOW WORKS CORRECTLY because `touchScrollEl` is the actual element the finger landed on

  <name>Task 2: Add Page Visibility API timer pause/resume (MJS-03)</name>
  <files>wolfond-report-2024-2026.html</files>
  <action>
**Add visibility change listener to pause/resume auto-advance timer when tab is hidden.**

The existing timer system uses:
- `autoTimer` — setInterval that fires at 120s, advances slide
- `countdownTimer` — setInterval that fires every 1s, decrements `secondsLeft` and updates display
- `startAutoAdvance()` — creates both intervals, sets `secondsLeft = AUTO_SECONDS`
- `stopAutoAdvance()` — clears both intervals, sets them to null
- `resetAutoAdvance()` — calls stop then start (called by `goToSlide()` at line ~1214)
- `secondsLeft` — tracks remaining seconds, decremented by `countdownTimer`

**Implementation — 2 additions:**

1. **Add `autoTimerPaused` flag near the existing timer variable declarations:**
   Find where `autoTimer`, `countdownTimer`, `secondsLeft` are declared (around lines 1096-1100). Add:
   ```javascript
   let autoTimerPaused = false;
   ```

2. **Add `visibilitychange` event listener after `stopAutoAdvance()` function (after around line 1132):**
   ```javascript
   document.addEventListener('visibilitychange', () => {
     if (document.hidden) {
       // Tab hidden — pause both intervals if running
       if (autoTimer !== null || countdownTimer !== null) {
         stopAutoAdvance();
         autoTimerPaused = true;
       }
     } else if (autoTimerPaused) {
       // Tab visible again — resume from remaining seconds
       autoTimerPaused = false;
       if (secondsLeft <= 0) {
         // Tab was hidden longer than timer duration — restart fresh
         startAutoAdvance();
       } else {
         // Resume countdown display from where it left off
         updateCounterDisplay();
         countdownTimer = setInterval(() => {
           secondsLeft--;
           if (secondsLeft < 0) secondsLeft = 0;
           updateCounterDisplay();
         }, 1000);
         // Fire slide advance after remaining seconds
         autoTimer = setInterval(() => {
           if (document.getElementById('videoModal').classList.contains('open')) return;
           if (current < slides.length - 1) {
             goToSlide(current + 1);
           } else {
             goToSlide(0);
           }
           // goToSlide() calls resetAutoAdvance() internally, which restarts fresh
         }, secondsLeft * 1000);
       }
     }
   });
   ```

**Key design decisions per user CONTEXT.md:**
- Resume from `secondsLeft` (remaining time), NOT reset to full 120s — user explicitly decided this
- Only react to `visibilitychange` (full tab hide), not `blur`/`focus` or partial occlusion
- `autoTimerPaused` flag distinguishes "system paused the timer because tab was hidden" from "user stopped auto-advance via some other mechanism" — prevents accidental resume
- `updateCounterDisplay()` called immediately on resume before the interval fires, so the display updates instantly (addresses Research open question #3)
- `secondsLeft <= 0` guard prevents double-advance when returning to a long-hidden tab (addresses Research Pitfall #3)
- No special return UI after long absence — timer was paused, so no unexpected slide advancement (per user decision)
